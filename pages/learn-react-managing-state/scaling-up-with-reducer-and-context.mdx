🔵 **Simple Explanation (Like an Instructor):**

- Sometimes, we want many components to **share** and **change** the same state.
- We can do this by **combining `useReducer` and Context API**.
- First, **make two contexts**:
  - One for **state** (to read data).
  - One for **dispatch** (to update/change data).
- In the top-level component, **use `useReducer`** and **provide** these two contexts to the children.
- Any child component can now **use** state and dispatch easily.
- To make it **even cleaner**, we can **put everything in one file**:
  - A `TasksProvider` component that provides the contexts.
  - Custom Hooks like `useTasks()` to read state, and `useTasksDispatch()` to update state.
- You can create **many** such reducer + context pairs for different features (like tasks, users, notifications, etc.)

---

🔵 **Real-Life Example:**

Imagine you are building a **To-Do App**.

- You want users to **see tasks** and also **add or delete tasks** from anywhere in your app (like Home page, Sidebar, Dashboard).
- Instead of passing state and functions manually (`props drilling`), you:
  - Create a **TasksContext** (for the list of tasks).
  - Create a **TasksDispatchContext** (for functions like addTask, deleteTask).
- You make a **TasksProvider** component at the top that wraps your app.
- In any child component, you just call `useTasks()` to get tasks, and `useTasksDispatch()` to modify them.
- Now your app is **organized**, **easy to scale**, and **no messy props passing**!

---

**রিয়েল লাইফ উদাহরণ:**

ভাবো, তুমি একটা অফিস ম্যানেজ করছো। অফিসে দুইটা গুরুত্বপূর্ণ কাজ আছে:

1. **নোটবুক রাখা** (সব কাজের লিস্ট রাখার জন্য)
2. **নোটবুকে কিছু লেখা বা কাটাকাটি করা** (নতুন কাজ যোগ করা, পুরাতন কাজ মুছে ফেলা)

এখন তুমি চাইছো অফিসের সবাই যেন সেই নোটবুক দেখে কাজ পড়তে পারে, আবার কাজ যোগ করতেও পারে।

তুমি কীভাবে করবে?

- **নোটবুক** হচ্ছে **state** (data যেখানে রাখা হয়েছে)।
- **কলম** হচ্ছে **dispatch function** (কাজ যোগ করা বা মুছে ফেলার টুল)।
- তুমি অফিসের সবাইকে নোটবুক আর কলম ব্যবহার করার অনুমতি দিচ্ছো।

এই জন্য তুমি দুইটা আলাদা permission বানালে:

- একটা permission শুধু **নোটবুক পড়ার জন্য** (state context)
- আরেকটা permission শুধু **কলম চালানোর জন্য** (dispatch context)

তারপর অফিসের যেই মানুষ চায় শুধু কাজ দেখবে, সে নোটবুক context নেবে।
আর যেই মানুষ কাজ যোগ বা মুছতে চায়, সে কলম context নেবে।

আর যদি চাও কাজটা আরো সুন্দরভাবে করতে, তুমি একটা **"OfficeManager"** বানাবে (একটা Provider), যেটা নতুন worker আসলে অটোমেটিক নোটবুক আর কলম দিয়ে দিবে — আলাদা করে permission নিতে হবে না।

আর worker যদি ভুলে যায় কীভাবে ব্যবহার করতে হয়, তুমি তার জন্য দুইটা **custom Hook** বানাবে:

- `useNotebook()` → কাজের লিস্ট পড়ার জন্য।
- `usePen()` → কাজ যোগ বা মুছার জন্য।

**এক লাইনে বলতে গেলে:**

> **Context আর Reducer মিলে অফিসের একটা সিস্টেম বানাচ্ছে, যেখানে সবাই ঠিকমতো কাজ পড়তে বা আপডেট করতে পারছে।**
